# 開発役割定義

## 1. 目的

本書は、Google Classroom運用支援AIエージェントのMVPを4名で実装するため、担当範囲、成果物、連携方法を定義する。

- AI班: `hinata`、`kimu`
- GUI班: `chiune`、`mocky`
- AI班とGUI班の境界は、`REQUIREMENTS.md`で定義されたAIアウトプットJSONとする。
- 担当者は自分の成果物について、実装、テスト、レビュー指摘への対応まで責任を持つ。

## 2. 役割分担

### 2.1 hinata — AI生成・品質責任者

AIによる文章生成と、生成結果の品質・安全性を担当する。

主な担当:

- タスク種別ごとのプロンプト設計
- リマインド文、課題要約、週次レポート、お知らせ文の生成処理
- 口調、出力形式、教師の追加指示を反映する生成ロジック
- 事実と推測の分離、ハルシネーション抑止、禁止事項の適用
- 個人情報を必要以上にプロンプトへ渡さないための制御
- AI生成失敗時のフォールバックと`AI_GENERATION_FAILED`の処理
- 生成文の品質テストと代表的な入力・期待出力の整備

主な成果物:

- プロンプトテンプレート
- AIモデル呼び出し処理
- 各`agentTaskType`の生成処理
- AI生成処理の単体テスト
- AI出力品質の確認用サンプル

担当外:

- 画面コンポーネントの実装
- Google OAuthおよびClassroomへの投稿処理
- PDFバイナリやGoogle Documentの実生成

### 2.2 kimu — AIデータ処理・JSON契約責任者

ClassroomデータをAI処理へ渡す前の整理と、GUIへ返す構造化JSONの正確性を担当する。

主な担当:

- Course、CourseWork、StudentSubmissionデータの正規化
- 未提出者、期限接近者、遅延提出者の判定ロジック
- AI処理のタスク振り分けと入力データ構築
- AIアウトプットJSON Schemaの定義とバージョン管理
- `summary`、`gui`、`outputs`、`approval`、`errors`の組み立て
- Markdown、PDF、Google Document用の構造化データ生成
- エラーコードの統一と、部分的なデータ取得失敗への対応
- JSON Schema検証、判定ロジック、出力データの単体テスト

主な成果物:

- AI入出力の型定義およびJSON Schema
- 提出状況の分類・集計処理
- GUIへ返すレスポンス生成処理
- 正常系・異常系のJSONサンプル
- AIデータ処理の単体テスト

担当外:

- 画面レイアウトやUI状態管理
- Google APIへの実リクエストとOAuth画面
- AI生成文の表現品質に関する最終判断

### 2.3 chiune — GUI画面・状態管理責任者

教師が情報を確認し、AIへ処理を依頼し、結果を編集できる画面を担当する。

主な担当:

- 画面構成、ルーティング、共通レイアウトの実装
- ログイン、コース選択、ダッシュボード、課題詳細画面の実装
- AI出力確認画面と編集可能フィールドの実装
- カード、表、警告、エラー表示の共通コンポーネント化
- ローディング、空データ、取得失敗、再試行の画面状態管理
- AIアウトプットJSONを画面表示用データへ変換する処理
- レスポンシブ対応、アクセシビリティ、基本的なUIテスト

主な成果物:

- 各画面と共通UIコンポーネント
- フロントエンドの状態管理
- AI出力JSONの表示・編集処理
- モックデータを使った画面テスト

担当外:

- プロンプトおよびAIモデル呼び出し処理
- 未提出者判定などの業務ロジック
- Classroomへの投稿、PDF、Google Documentの実生成

### 2.4 mocky — GUI連携・承認・出力責任者

GUIから利用するGoogle連携と、教師の承認後に実行される外部操作を担当する。

主な担当:

- Google OAuth 2.0ログインと必要最小限のスコープ管理
- Google Classroom APIによるコース、課題、提出状況の取得
- 投稿確認画面と教師承認フローの実装
- 承認済みClassroomお知らせの投稿処理
- Markdown保存、PDF生成、Google Document作成処理
- API失敗、認証期限切れ、権限不足、レート制限の処理
- OAuthトークン、ログ、個人情報の安全な取り扱い
- Google API連携と出力処理の結合テスト

主な成果物:

- Google認証・Classroom API連携処理
- 投稿確認および承認アクション実行処理
- Markdown、PDF、Google Documentの出力処理
- Google APIのモックと結合テスト

担当外:

- AIプロンプトと生成文の品質調整
- AIアウトプットJSON Schemaの単独変更
- ダッシュボードなど一般画面の最終的なUI設計

## 4. AI班とGUI班の連携規則

### 4.1 JSON契約

- `schemaVersion`を必須とし、MVPでは`1.0.0`を使用する。
- kimuはJSON Schemaと正常系・異常系サンプルを管理する。
- chiuneとmockyは、実装前にサンプルJSONで表示・操作を確認する。
- 必須フィールドの追加、削除、型変更は破壊的変更として扱う。
- 破壊的変更は4名で影響を確認し、`schemaVersion`を更新してから実装する。
- GUI側は未知のフィールドを無視し、AI側は必須フィールドを省略しない。

### 4.2 データの受け渡し

1. mockyがGoogle Classroomからデータを取得する。
2. kimuが取得データを正規化し、提出状況を分類する。
3. hinataが必要な文章を生成する。
4. kimuが生成結果を共通JSONへ組み立て、検証する。
5. chiuneがJSONを表示し、教師が編集できる状態にする。
6. mockyが教師の明示的な承認後に投稿または出力する。

### 4.3 安全性

- 教師の承認なしにClassroomへ投稿しない。
- 生徒の個人情報は、処理に必要な範囲だけ保持・送信する。
- 生徒向け投稿に、他の生徒の提出状況や氏名を含めない。
- ログへOAuthトークン、生成プロンプト内の個人情報、投稿対象者一覧をそのまま記録しない。
- AIの提案とClassroom APIから取得した事実を区別して表示する。

## 5. 開発・レビュー規則

- 各機能は主担当が実装し、表に記載した協力者がレビューする。
- AI班はGUI実装を待たず、JSONサンプルとモックを先に提供する。
- GUI班はAI実装を待たず、合意済みJSONサンプルで画面を実装する。
- 結合前に、JSON Schema検証、主要画面、未提出者判定、承認フローの単体テストを通す。
- 結合テストでは「データ取得 → AI処理 → GUI編集 → 教師承認 → 投稿・出力」を確認する。
- 担当境界を越える変更は、該当箇所の主担当へレビューを依頼する。

## 6. MVP完了時の共同責任

4名は、以下が一連の操作として成立することを共同で確認する。

1. 教師がGoogleアカウントでログインできる。
2. コース、課題、提出状況を取得できる。
3. 未提出者を正しく抽出できる。
4. AIがリマインド文を構造化JSONで返せる。
5. GUIで内容を表示・編集できる。
6. 教師の承認なしにはClassroomへ投稿されない。
7. 承認後にClassroomへ投稿できる。
8. MarkdownおよびPDFとして出力できる。
